# /commit コマンド

GitHub運用ルールに従ってコミットを作成します。

## 手順

1. `git status` で変更内容を確認
2. `git diff --staged` でステージされた変更を確認（なければ `git diff`）
3. 変更内容を分析して適切なコミットメッセージを生成
4. ユーザーに確認を求める
5. コミットを実行

## コミットメッセージ形式

```
<type>: <subject>

<body>（任意）

Co-Authored-By: Claude Opus 4.5 <noreply@anthropic.com>
```

## Type 一覧

| Type | 説明 | 使用場面 |
|------|------|----------|
| `feat` | 新機能 | 新しい機能を追加した時 |
| `fix` | バグ修正 | バグを修正した時 |
| `docs` | ドキュメント | ドキュメントのみの変更 |
| `refactor` | リファクタリング | 機能変更を伴わないコード改善 |
| `test` | テスト | テストの追加・修正 |
| `chore` | 雑務 | ビルド設定、依存関係の更新等 |
| `style` | スタイル | コードスタイルの修正（空白、フォーマット等） |
| `perf` | パフォーマンス | パフォーマンス改善 |

## ルール

- subject は日本語で簡潔に（50文字以内推奨）
- body は変更の理由や詳細を記載（任意）
- 1コミット = 1つの論理的な変更
- 動作する状態でコミット

## 実行例

```
feat: メール通知機能を追加

SMTPサーバーを使用してファイル処理完了時に
通知メールを送信する機能を実装。

Co-Authored-By: Claude Opus 4.5 <noreply@anthropic.com>
```
