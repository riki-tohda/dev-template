# /commit コマンド

GitHub運用ルールに従ってコミットを作成します。

## 手順

1. `git status` で変更内容を確認
2. `git diff --staged` でステージされた変更を確認（なければ `git diff`）
3. 更新履歴ファイルを作成（大規模変更の場合）
4. 変更内容を分析して適切なコミットメッセージを生成
5. ユーザーに確認を求める
6. コミットを実行

## コミットメッセージ形式

```
<type>: <subject>

<body>（任意）

Co-Authored-By: Claude Opus 4.5 <noreply@anthropic.com>
```

## Type 一覧

| Type | 説明 | 使用場面 |
|------|------|----------|
| `feat` | 新機能 | 新しい機能を追加した時 |
| `fix` | バグ修正 | バグを修正した時 |
| `docs` | ドキュメント | ドキュメントのみの変更 |
| `refactor` | リファクタリング | 機能変更を伴わないコード改善 |
| `test` | テスト | テストの追加・修正 |
| `chore` | 雑務 | ビルド設定、依存関係の更新等 |
| `style` | スタイル | コードスタイルの修正（空白、フォーマット等） |
| `perf` | パフォーマンス | パフォーマンス改善 |

## ルール

- subject は日本語で簡潔に（50文字以内推奨）
- body は変更の理由や詳細を記載（任意）
- 1コミット = 1つの論理的な変更
- 動作する状態でコミット

## 更新履歴の作成ルール

変更規模に応じて `docs/更新履歴/` にドキュメントを作成する。

| 変更規模 | 記録先 | 例 |
|---------|--------|-----|
| 大 | 個別ファイル（`YYYY-MM-DD_内容.md`） | 新機能追加、API変更、DB変更 |
| 中 | 個別ファイル | バグ修正（複数ファイル）、リファクタリング |
| 小 | 月次ファイル（`YYYY-MM_微修正.md`） | typo修正、スタイル微調整、単一ファイルの小修正 |
| 極小 | コミットメッセージのみ | コメント追加、空白調整 |

### 個別ファイルの記載内容

```markdown
# タイトル

## 概要
変更の目的と概要

## 実装内容
### 変更ファイル一覧
| ファイル | 変更内容 |
|---------|---------|

### 削除ファイル一覧（該当する場合）
### 新規ファイル一覧（該当する場合）

## 影響範囲
## 関連ドキュメント
```

## 実行例

```
feat: メール通知機能を追加

SMTPサーバーを使用してファイル処理完了時に
通知メールを送信する機能を実装。

Co-Authored-By: Claude Opus 4.5 <noreply@anthropic.com>
```
